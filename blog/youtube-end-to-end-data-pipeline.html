From API to Dashboard: Building an End-to-End YouTube Data Pipeline

July 1, 2026 · 7 min read · Data Engineering, Airflow, PostgreSQL

In content creation and digital marketing, visibility is everything. However, relying on default platform analytics dashboards limits the depth of insights. To truly understand audience growth, content performance, and engagement trends over time, you need programmatic access to the underlying data. That realization is what drove me to build a containerized ELT (Extract, Load, Transform) data pipeline designed to ingest, process, and analyze YouTube channel metrics at scale.

Using the official YouTube Data API (v3) as a source, the platform orchestrates extraction tasks through Apache Airflow, stores and models the records inside PostgreSQL using layered schemas, ensures data quality via Soda Core validation tests, and is packaged cleanly using Docker Compose. Finally, a robust GitHub Actions CI/CD workflow manages testing and deployment. Here is a deep dive into the architecture, challenges, and implementation details of the pipeline.

The Architecture: Designing a Layered ELT Pipeline

Rather than building a brittle script that pulls data and maps it straight to a single dashboard, this project implements a structured ELT architecture. This keeps extraction lightweight, preserves historical data, and builds staging and analytics marts on a schedule.

Fig. 1 — End-to-End Data Flow Architecture

As illustrated in the diagram, the data flow progresses from left to right. The YouTube API serves raw JSON payloads representing video analytics, play statistics, comments, and channel updates. These are pulled by Airflow Tasks and loaded directly into PostgreSQL's RAW schema (equivalent to the Bronze layer). From there, SQL transformations structure the data into the STAGING (Silver) and MARTS (Gold) schemas. Quality verification is managed at boundaries using Soda Core check constraints.

Airflow Orchestration: Scheduling and DAG Design

Apache Airflow is the backbone of this pipeline, responsible for managing task dependencies, handles API rate limits with retry mechanisms, and logs errors cleanly. Rather than deploying a single, monolith DAG, the workflow is split logically to enable granular execution schedules:

  • Daily Video Metrics DAG: Runs daily at midnight to capture fast-moving statistics like view count, likes, comment count, and share parameters.
  • Weekly Channel & Comments DAG: Runs every Monday to update broad channel status profiles, subscriber growth trends, and fetch high-volume historical comment threads.

A typical execution graph involves extracting data from the YouTube API, storing it temporarily as flat Parquet files to prevent memory issues, importing those files into PostgreSQL, running data quality tests, and then triggering the SQL models that refresh down-stream tables.

Fig. 2 — Airflow Task Dependencies

By enforcing the Soda validation step check_raw_soda before running the SQL queries (transform_stage), the pipeline acts as a self-healing system. If an upstream API payload format changes unexpectedly (e.g. YouTube deprecating a field or returning null values), the DAG halts execution immediately, avoiding downstream corruption.

PostgreSQL Schema Layering: Bronze, Silver, and Gold in Postgres

While Snowflake and Databricks dominate enterprise data warehousing, you can build a highly effective, low-cost structured warehouse architecture directly in a single PostgreSQL instance using logical schemas. In this pipeline, the layers are structured as follows:

1. Raw Schema (Bronze)

This layer holds the incoming JSON documents exactly as fetched from the YouTube Data API. We use Postgres's native JSONB data type to store the raw payload along with some metadata columns:

CREATE TABLE raw.youtube_videos (
    id SERIAL PRIMARY KEY,
    video_id VARCHAR(50) NOT NULL,
    raw_payload JSONB NOT NULL,
    extracted_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

2. Staging Schema (Silver)

The staging schema parses the raw JSON payloads into clean, tabular formats. It handles data type casting, converts timestamps, flattens arrays, and deduplicates the records. We implement this using a combination of SQL views and physical tables populated through incremental insertions:

CREATE TABLE staging.stg_videos AS
SELECT 
    video_id,
    (raw_payload->'snippet'->>'title')::TEXT as title,
    (raw_payload->'snippet'->>'publishedAt')::TIMESTAMP WITH TIME ZONE as published_at,
    (raw_payload->'statistics'->>'viewCount')::BIGINT as view_count,
    (raw_payload->'statistics'->>'likeCount')::BIGINT as like_count,
    extracted_at
FROM raw.youtube_videos;

3. Marts Schema (Gold)

Finally, the marts schema models business metrics. Here, facts and dimensions are modeled to support fast dashboards. Dimensions are slowly changing (like channel profile updates or video tags), while facts represent daily snapshots of views, subscriptions, and engagement indices.

Data Quality Verification with Soda Core

In data engineering, silent failures are the worst. A script that runs successfully but inputs zero rows or corrupts dates is more dangerous than a crashing script. To catch these issues, the pipeline integrates Soda Core checks directly inside the Airflow sequence.

Soda Core is a lightweight YAML-defined assertions framework. Before transitioning data from staging to the marts layer, Airflow executes a Soda check against the tables. A configuration file like the one below validates the video records:

# soda/checks/staging_videos.yml
checks for staging.stg_videos:
  - row_count > 0
  - video_id:
      text_length_between: [1, 50]
      missing_percent = 0%
  - view_count:
      min >= 0
  - published_at:
      invalid_percent = 0%
  - duplicate_count(video_id) = 0

If any test fails, Soda returns a non-zero exit code, raising an Airflow exception. This ensures that only verified, clean datasets reach the final analysis layer.

Local Development Environment and CI/CD

A key goal was making the pipeline easy to run locally while keeping it reproducible in production. We accomplished this using Docker Compose and GitHub Actions.

Docker Compose

The local development environment runs Airflow, PostgreSQL, and Soda Core in isolated containers. By mapping local directories to container workspaces, we can write DAGs and SQL code locally and see updates immediately:

  • postgres:15-alpine: Houses our multi-layer database.
  • apache/airflow:2.7-python3.10: Configured with necessary environment connections to access the database and the API.
  • local-runners: A utility container pre-configured with Python libraries and Soda Core CLI for running integration checks manually.

GitHub Actions CI/CD Pipeline

Whenever a pull request is created or code is pushed to main, GitHub Actions automatically executes our pipeline tests. The workflow provisions a mock PostgreSQL database, pulls test JSON data, performs the raw-to-staging transformations, runs the Soda Core checks, and runs unit tests using pytest. Only when all tests pass is the code merged.

Challenges and Solutions

1. YouTube API Quotas

The YouTube Data API has a strict daily quota (usually 10,000 units). Operations like listing video comments are expensive. To solve this, the pipeline caches API results locally in Airflow's filesystem during extraction and runs incrementally, querying only videos updated in the last 48 hours.

2. Schema Evolution

If YouTube introduces a new field or nested structure to the payload, strict tables might crash. By using the JSONB type in PostgreSQL's Raw layer, we avoid load crashes. The parsing rules can then be safely updated in the staging views without losing historical raw records.

Conclusion

Building this end-to-end pipeline highlights the importance of data quality gates and schema separation. Designing with Docker and Airflow from day one ensures scalability. The architecture is flexible enough to integrate other social media APIs (like Twitch or Twitter) by simply adding a new raw schema and extending the transformations.
