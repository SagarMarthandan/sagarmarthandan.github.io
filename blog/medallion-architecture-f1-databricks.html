Medallion Architecture in Practice: Building an F1 Data Platform on Azure Databricks

July 8, 2026 · 8 min read

Formula 1 is a sport driven by millisecond advantages and massive amounts of telemetry data. From lap times and pit stop strategies to driver telemetry and qualifying performance spanning over seven decades (1950 to present), raw motor racing datasets present classic data engineering challenges: heterogeneous file formats, schema evolution, nested structures, and concurrent analytical reads.

When building a cloud-native platform to analyze Formula 1 historical race performance, simply dropping raw CSVs and JSON payloads into object storage leads to "data swamp" symptoms: inconsistent data types, duplicate records, missing race telemetry, and painfully slow analytics queries. To solve this, I designed and implemented an end-to-end data platform using the Medallion Architecture pattern on Azure Databricks, powered by PySpark, Delta Lake, and Azure Data Factory (ADF).

Here is an engineering teardown of how the platform transforms raw motor racing feeds into high-performance dimensional models for downstream analytics.

What is Medallion Architecture and Why It Matters

The Medallion Architecture is a data design pattern that structures data logically into three distinct layers — Bronze, Silver, and Gold. Each layer progressively improves data quality, structure, and query readiness:

  • Bronze Layer (Raw Storage): The landing zone. Ingests raw data from source systems into Azure Data Lake Storage Gen2 (ADLS Gen2) as-is (CSV, JSON, Parquet) with append-only semantics and metadata lineage (such as ingestion timestamps and source file paths).
  • Silver Layer (Cleansed & Conformed): The enterprise data lake layer. Raw Bronze payloads are parsed, cleansed, deduplicated, type-cast, and converted into ACID-compliant Delta Lake tables with explicit schemas.
  • Gold Layer (Curated Business Analytics): The consumption layer. Data is aggregated and modeled into Star Schemas (fact and dimension tables) and analytical views optimized for BI tools like Power BI, Superset, or custom analytics applications.

The F1 Dataset & Ingestion Sources

The platform consumes motor racing data originating from the Ergast API dataset. The dataset comprises eight core entity streams:

  1. circuits (CSV) — Circuit names, locations, country, latitude/longitude, and altitude.
  2. races (CSV) — Grand Prix race schedules, rounds, dates, times, and circuit links.
  3. drivers (JSON) — Driver identities, dob, nationality, code, and nested name objects.
  4. constructors (JSON) — Team details, constructor refs, and nationalities.
  5. results (CSV) — Race results including grid position, final position, points earned, laps completed, and time differentials.
  6. pit_stops (Multi-line JSON) — Pit stop durations, lap numbers, stop counts, and millisecond timings.
  7. lap_times (Split CSVs) — Full lap-by-lap telemetry for every driver across all races.
  8. qualifying (Multi-line JSON) — Q1, Q2, and Q3 lap times per driver per Grand Prix.

Because these feeds mix flat CSVs with nested multi-line JSON structures, attempting direct SQL analytical queries without a structured lakehouse model introduces severe performance bottlenecks and fragile transformations.

Bronze Layer: ADLS Gen2 Ingestion with ADF

In the Bronze layer, raw files are ingested into Azure Data Lake Storage Gen2 (ADLS Gen2) using Azure Data Factory. To preserve auditing capabilities and data lineage, raw payloads are never modified during Bronze ingestion.

Databricks mounts ADLS Gen2 storage containers using Azure Service Principals registered in Entra ID (formerly Azure Active Directory), with secret credentials secured inside Azure Key Vault secret scopes. This avoids hardcoding keys in PySpark notebooks:

client_secret = dbutils.secrets.get(scope="f1-keyvault-scope", key="databricks-app-client-secret")

configs = {
    "fs.azure.account.auth.type": "OAuth",
    "fs.azure.account.oauth.provider.type": "org.apache.hadoop.fs.azurebfs.oauth2.ClientCredsTokenProvider",
    "fs.azure.account.oauth2.client.id": "4b68xxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "fs.azure.account.oauth2.client.secret": client_secret,
    "fs.azure.account.oauth2.client.endpoint": "https://login.microsoftonline.com/your-tenant-id/oauth2/token"
}

dbutils.fs.mount(
    source="abfss://raw@f1dlstorage.dfs.core.windows.net/",
    mount_point="/mnt/f1dlstorage/raw",
    extra_configs=configs
)

When ingesting raw files into Bronze, we attach an ingestion timestamp (ingestion_date) and dynamic source path tracking to record exactly when and where each file entered the lakehouse.

Silver Layer: Schema Enforcement & Delta Cleansing

The Silver layer transitions raw Bronze datasets into clean, reliable Delta Lake tables. The key engineering goals here are schema enforcement, data type normalization, column renaming to snake_case, and handling null or invalid telemetry values.

Below is a PySpark snippet transforming raw nested driver JSON payloads into the Silver Delta table:

from pyspark.sql.types import StructType, StructField, IntegerType, StringType, DateType
from pyspark.sql.functions import col, current_timestamp, concat, lit

# Define explicit schema to guarantee schema enforcement
name_schema = StructType([
    StructField("forename", StringType(), True),
    StructField("surname", StringType(), True)
])

drivers_schema = StructType([
    StructField("driverId", IntegerType(), False),
    StructField("driverRef", StringType(), True),
    StructField("number", IntegerType(), True),
    StructField("code", StringType(), True),
    StructField("name", name_schema, True),
    StructField("dob", DateType(), True),
    StructField("nationality", StringType(), True),
    StructField("url", StringType(), True)
])

# Read raw JSON from Bronze container
raw_drivers_df = spark.read \
    .schema(drivers_schema) \
    .json("/mnt/f1dlstorage/raw/drivers.json")

# Cleanse, flatten nested fields, rename to snake_case, add metadata
silver_drivers_df = raw_drivers_df \
    .withColumnRenamed("driverId", "driver_id") \
    .withColumnRenamed("driverRef", "driver_ref") \
    .withColumn("name", concat(col("name.forename"), lit(" "), col("name.surname"))) \
    .withColumn("ingestion_date", current_timestamp()) \
    .drop("url")

# Write to Silver Delta Lake table with partition overwrite or merge
silver_drivers_df.write \
    .format("delta") \
    .mode("overwrite") \
    .saveAsTable("f1_silver.drivers")

By enforcing StructType schemas at read time, corrupted payloads or unexpected data shape shifts fail immediately in the Silver ingestion step rather than contaminating downstream Gold reporting tables.

Gold Layer: Dimensional Modeling for F1 Analytics

The Gold layer models cleansed Silver tables into a business-ready Star Schema. In Formula 1 analytics, race outcomes depend heavily on multi-dimensional relationships: driver performance over time, constructor championship points, circuit characteristics, and pit stop strategy efficiency.

Gold Data Model Structure:

  • fact_results — Granular race results (points, positions, grids, fastest laps, durations).
  • fact_lap_times — High-volume fact table storing lap-by-lap milliseconds per driver.
  • dim_drivers — Driver dimension (name, nationality, racing code, dob).
  • dim_circuits — Circuit location dimension (lat, long, country, altitude).
  • dim_constructors — Constructor dimension (team name, nationality).
  • driver_standings — Aggregated analytical Gold view computing cumulative championship rankings by season.

Here is how PySpark window functions compute the aggregate Gold table for driver standings across race seasons:

from pyspark.sql.window import Window
from pyspark.sql.functions import sum as _sum, count, when, col, rank

# Calculate total points and wins per driver per race season
race_results_df = spark.table("f1_gold.fact_race_results")

driver_standings_df = race_results_df \
    .groupBy("race_year", "driver_name", "driver_nationality", "team_name") \
    .agg(
        _sum("points").alias("total_points"),
        count(when(col("position") == 1, 1)).alias("wins")
    )

# Apply Window spec to rank drivers by points and wins per year
driver_rank_spec = Window.partitionBy("race_year").orderBy(col("total_points").desc(), col("wins").desc())

final_standings_df = driver_standings_df \
    .withColumn("rank", rank().over(driver_rank_spec))

final_standings_df.write \
    .format("delta") \
    .mode("overwrite") \
    .saveAsTable("f1_gold.driver_standings")

Delta Lake Power Features: ACID, Time Travel & Schema Evolution

Building on standard Parquet files leaves data pipelines vulnerable to partial write failures during job restarts and concurrent read conflicts. Using Delta Lake as the storage format across Silver and Gold provides three key production guarantees:

1. ACID Transactions & Atomic Upserts

When updating lap times or re-processing race telemetry files, Delta Lake uses a transaction log (_delta_log) to guarantee atomic commits. We perform idempotent upserts into Silver tables using PySpark's DeltaTable.merge() API:

from delta.tables import DeltaTable

silver_delta_table = DeltaTable.forName(spark, "f1_silver.results")

silver_delta_table.alias("target").merge(
    source=incoming_results_df.alias("updates"),
    condition="target.result_id = updates.result_id AND target.race_id = updates.race_id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()

2. Time Travel & Historical Auditability

FIA penalty decisions and point recalculations frequently alter official race results days after a Grand Prix concludes. Delta Lake's time travel enables auditing past table states for historical reproducibility:

# Query driver standings exactly as they were recorded at Version 3
df_version_3 = spark.read \
    .option("versionAsOf", 3) \
    .table("f1_gold.driver_standings")

# Or restore table to prior state after erroneous ingest
spark.sql("RESTORE TABLE f1_silver.results TO VERSION AS OF 2")

3. Controlled Schema Evolution

When Formula 1 introduced Sprint Races, raw qualifying data payloads expanded to include sprint shootout attributes. Setting .option("mergeSchema", "true") allowed Delta Lake to gracefully append new columns without breaking legacy downstream readers.

Azure Data Factory Orchestration Pattern

To run the Medallion pipeline in production, Azure Data Factory (ADF) orchestrates notebook execution sequentially across layers while enforcing data validation gates between stages.

Fig. 2 — Azure Data Factory Orchestration Pipeline Flow

The pipeline relies on dynamic parameter passing. When ADF triggers a Databricks notebook activity, it passes p_file_date dynamically. In PySpark, widgets capture parameter values seamlessly:

dbutils.widgets.text("p_file_date", "2024-03-24")
v_file_date = dbutils.widgets.get("p_file_date")

# Filter incoming Bronze files dynamically by run date
filtered_df = raw_results_df.filter(col("file_date") == v_file_date)

Production Lessons & Performance Tuning

Building this platform surfaced critical lessons regarding Spark execution performance and storage optimization:

1. Partitioning Strategy & Small File Mitigations

Partitioning Gold tables by race_year dramatically sped up historical queries (e.g., querying 2023 season driver standings skipped scanning 70+ years of unneeded Parquet files). However, over-partitioning Silver tables led to thousands of tiny 50KB files. We scheduled Databricks OPTIMIZE jobs with Z-ORDER BY (race_id, driver_id) to consolidate file sizes to 128MB chunks and co-locate related records for fast predicate pushdowns.

2. Single-Node vs Multi-Node Compute Costs

For daily incremental updates, spinning up multi-node auto-scaling clusters was cost-inefficient. Leveraging single-node Databricks clusters for Silver transformation jobs reduced Azure DBU consumption by ~40% while remaining well within SLA targets.

3. Lakehouse Governance Transition

While mounting ADLS Gen2 directly via DBFS is straightforward during initial development, modern Databricks deployments benefit from Unity Catalog to provide fine-grained table access control, automated column lineage, and centralized data discovery across Azure environments.
